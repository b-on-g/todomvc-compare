# Что всё таки не так с веб компонентами

Господа, [продолжаем разбираться](https://habr.com/ru/articles/1019206/) в тонкостях веб компонент. Сделал тут [бенч — сравнение](https://github.com/b-on-g/todomvc-compare) фреймворков ($mol/Lit/Symbiote) по TodoMVC. Вроде говорим об одном, а бенч о другом, разве не так? Ан-нет — чтобы разобраться с веб компонентами нужны фреймворки которые ставят их во главу угла, те, кто "сделал на них ставку".

Вот что мне удалось понять:

**Первое. Память:** 124 байта на веб-компонент, и 16 байт на JS object. Разница на порядок — это много, и без виртуализации интерфейс скорее всего будет лагать.

```javascript
// Lit: каждый todo-item — это HTMLElement (C++ heap, ~124 байт minimum)
@customElement("todo-item")
export class TodoItem extends LitElement { ... }
// <todo-item> сразу аллоцирует DOM-ноду при createElement

// Symbiote: аналогично, каждый <todo-item> = HTMLElement
class TodoItem extends Symbiote { ... }
TodoItem.reg('todo-item');

// $mol: компонент — JS-объект. DOM создаётся ТОЛЬКО при рендере.
// 1000 задач в модели ≠ 1000 DOM-элементов
// $mol рендерит только видимые компоненты (виртуализация)
task_rows() {
    return this.task_ids_filtered().map(id => this.Task_row(id))
}
```

| Custom Element                      | js/mol_view        |
| ----------------------------------- | ------------------ |
| 1 000 задач: ~124 KB только на ноды | ~16 KB на объекты  |
| 10 000 задач: ~1.2 MB              | ~160 KB            |

И это только базовые ноды — без текстовых, атрибутов, стилей и event listeners, которые тоже аллоцируются в C++ heap. Спецификация Custom Elements требует `class MyEl extends HTMLElement`. Нельзя создать CE без DOM-ноды.

Вот [тут](https://nolanlawson.com/2024/09/28/web-components-are-okay/) разбирается этот же довод от автора SolidJS. Далее ответ на довод от автора статьи:

> "If your goal is to build the absolute fastest framework you can, then you want to minimize DOM nodes wherever possible. This means that web components are off the table."

Проще говоря — хотите производительности? Не используйте веб компоненты.

**Второе**, помимо увеличения потребления памяти, мы теряем JIT оптимизацию.

| Операция                           | Время      | Во сколько раз  |
| ---------------------------------- | ---------- | --------------- |
| obj.title = x (JS-объект)          | ~1–2 ns    | эталон — 1x     |
| element.textContent = x (DOM)      | ~30–60 ns  | в 30x раз хуже  |
| element.setAttribute('class', x)   | ~50–100 ns | в 50x           |
| element.style.color = x            | ~80–150 ns | в 80x           |

Задача: выделить всё (актуально для почты например, которая не может за раз удалить все письма, если они не помещаются на страницу). Деградирует так:

| Задач   | Lit                                                  | $mol  | Разница |
| ------- | ---------------------------------------------------- | ----- | ------- |
| 100     | 60 µs (микросекунд)                                 | 3 µs  | 20x     |
| 1 000   | 600 µs                                               | 8 µs  | 75x     |
| 10 000  | 8 ms                                                 | 12 µs | 650x    |
| 100 000 | 120 ms (лаг — если кадр занимает больше 16 мс)      | 15 µs | 8 000x  |

И такой вопрос к читателям. Стоит ли вообще ориентироваться на [js-framework-benchmark](https://github.com/krausest/js-framework-benchmark) в таком случае? Мне кажется что нет. Не стоит рендерить то, что не видно. Там борются за спички, а все знают что на спичках экономить не нужно. Ну и вспомним цитату про perf выше.

Ну и пример в коде. Тут мы ещё и html парсим... (это плохо)

```javascript
// Lit: обновление через DOM property
// element.textContent = newValue  ← C++ binding, slow path
render() {
    return html`<label> ${this.text} </label>`;
    // Lit под капотом делает: node.textContent = value
}

// $mol: обновление через JS property
// this.title() — чтение из memo-кеша (JS heap)
// DOM обновляется батчем через requestAnimationFrame
task_title(id, next?) {
    return this.task(id, ...)!.title ?? ''
}
```

Идём дальше. WC (Web Components) используют push семантику для реактивности. Pull — думаю вряд ли будут поддерживать. На что это влияет? На количество строк кода **написанных вами**. А всем людям свойственно допускать ошибки — не даром есть поговорка: "Лучший код тот — который не написан". Смотрим на [бенч](https://github.com/b-on-g/todomvc-compare).

![Сравнение бенчмарков](https://habrastorage.org/r/w1560/getpro/habr/upload_files/d2a/7cf/fd2/d2a7cffd29174443b049d6c41898efae.png)
_Солидно._

Ну и пример в коде, для наглядности.

```javascript
// Lit: Push через EventTarget + requestUpdate
export class Todos extends EventTarget {
    #notifyChange() {
        this.dispatchEvent(new Event("change")); // push!
    }
    add(text) {
        this.#todos.push({ ... });
        this.#notifyChange(); // ← ручной push
    }
}
// Каждый компонент подписывается через @updateOnEvent("change")
// При ЛЮБОМ изменении ВСЕ подписчики получают уведомление

// Symbiote: Push через EventTarget аналогично
store.addEventListener('change', () => this.#render());

// $mol: Pull — автоматический граф зависимостей
@$mol_mem
groups_completed() {
    // Автоматически отслеживает зависимости:
    // task_ids() → task() → groups
    // Пересчитывается ТОЛЬКО когда изменились зависимости
    for (let id of this.task_ids()) {
        var task = this.task(id)!;
        groups[String(task.completed)].push(id);
    }
    return groups;
}
// Не нужен ни EventTarget, ни ручные подписки
```

Не зря Vue.js выбрал pull реактивность.

Коснёмся тестируемости. Чтобы протестировать веб компонент — его нужно отрендерить. Максимально неэффективно. Думаю, все мы не любим когда тесты выполняются долго.

Смотрим пример:

```javascript
// $mol: тесты БЕЗ DOM. Компонент — просто объект.
'task add'($) {
    const app = $hyoo_todomvc.make({ $ })
    app.Add().value('test')
    app.Add().submit()
    $mol_assert_equal(app.task_rows().at(-1)!.title(), 'test')
}
// Никакого document.createElement, никакого connectedCallback
// Просто вызовы методов и проверка состояния

// Lit: для тестирования нужен DOM
const el = document.createElement('todo-app');
document.body.appendChild(el); // connectedCallback!
await el.updateComplete; // ждём рендер!
el.shadowRoot.querySelector('.new-todo')... // доступ через Shadow DOM!

// В hyoo_todomvc есть 140 строк тестов.
// В Lit и Symbiote реализациях — 0. (а кода написано даже больше)
```

Про наследование.

Как это в Lit на тех самых "удобных маленьких веб компонентах":

```javascript
class TodoApp extends LitElement {
	render() {
		// Весь шаблон — монолитный блок. 40+ строк HTML.
		return html`
			<header>...</header>
			<section>
				<ul>
					${this.todos.map(t => html`...`)}
				</ul>
			</section>
			<footer>
				<span>...</span>
				<ul>
					...
				</ul>
				<button>...</button>
			</footer>
		`
	}
}

// Хочешь изменить только footer?
class MyTodoApp extends TodoApp {
	render() {
		// Нельзя сказать "возьми render() родителя, замени footer."
		// Единственный вариант — скопировать ВСЕ 40 строк
		// и поменять нужный кусок.
		return html`
			<header>...</header>
			// копия
			<section>...</section>
			// копия
			<footer>МОЙ ФУТЕР</footer>
			// всё ради этого
		`
	}
}
```

Как это в $mol:

```javascript
// Базовый компонент: прокручиваемая область
$mol_scroll
    sub /           ← контент
    scroll_top 0    ← позиция скролла
    event_scroll     ← обработчик

// TodoMVC НАСЛЕДУЕТ $mol_scroll и заменяет только sub (вложенные элементы):
$hyoo_todomvc $mol_scroll
    sub /
        <= Page $mol_list ...

// Скролл, позиция, обработчик — всё досталось бесплатно.
// Переопределили ТОЛЬКО содержимое.

// Можно пойти глубже — унаследовать сам $hyoo_todomvc и заменить, например, только футер:
$my_custom_todo $hyoo_todomvc
    foot <= My_footer $mol_view
// Всё остальное (список, ввод, фильтры) — наследуется как есть.
```

Аналогично с наследованием логики и стилей (уникально каскадно). 0 копипасты и boilerplate.

И под конец хотел бы ещё поспорить с одним утверждением. Далее моя вольная цитата из текста [@i360u](https://habr.com/ru/users/i360u/) ([первая ссылка](https://habr.com/ru/articles/1019206/) в статье):

> "Любое инженерное решение — это компромисс. Само по себе, наличие «против» не перевешивает все возможные «за», априори."

В какой-то степени с ним можно согласиться. Если бы не одно "НО". У нас у всех стоит одна задача — разработать максимально удобное, быстрое, красивое приложение, без легаси, чистым кодом, который легко поддерживать, и все такое. Ещё раз — задача у всех одна. И надо смотреть кто её лучше всего решает **в совокупности** всего frontend мира. Я знаю одно такое решение.

Это $mol — всё остальное хуже. Это факт.

Все эти "вендор лок", "страшно непонятно изучать новое" — это детские отмазки, чтобы не признавать факты. Везде в программировании надо изучать новое. А вендор лока вообще нет. Думайте. Кирилл. Подпишись.

## FAQ (из комментариев)

**"А если использовать виртуализацию — те CE ноды тоже не будут рендериться, разве не одно и то же?"**

Не совсем. С виртуализацией сложность переносится на контейнер выше. И на практике это [не так просто](https://github.com/nickmessing/todomvc-compare) — в $mol виртуализация встроена по умолчанию, а в WC-фреймворках её приходится проектировать самостоятельно. Каждое дополнительное архитектурное решение — это ещё один шанс ошибиться.

**"Для композиции — просто разбей render на renderHeader / renderBody / renderFooter и переопредели один в подклассе. Этому паттерну 40 лет."**

Полностью согласен. Суть скорее в том, что $mol обеспечивает это как **архитектурное ограничение** — разработчик физически не может породить тот монолитно-шаблонный антипаттерн, который я показал. Чем меньше ошибок может допустить программист и чем больше ловит компилятор — тем лучше. Опять же, дело в том, чтобы не думать и не писать "лишний" код.

**"Ты обобщаешь слишком радикально. Проблема не в веб компонентах как таковых — а в том, как их используют. Наивный рендер тысяч элементов через DOM упрётся в лимиты вне зависимости от подхода."**

Я тоже так думаю. Но многие реально смотрят на [js-framework-benchmark](https://github.com/krausest/js-framework-benchmark) и считают, что он отражает реальную производительность. Это вызывает фрустрацию — ощущение, что бенчмарк вводит в заблуждение. Если бенчмарк плохой, зачем он нужен? Мне кажется, надо использовать все доступные трюки и показать реальный максимум производительности JS. Он тоже не бесконечен, но докуда может дотянуться? Никто не знает — все застряли на HTML. По моим оценкам, на $mol можно написать целую ОС, и она будет быстрее и стабильнее многих существующих. (И нотификации будут приходить мгновенно с обновлением колокольчика.)

**"А как же Ctrl+F поиск по странице? Нельзя найти то, что не отрендерено."**

Справедливо. Но забавно, что в $mol поиск как раз работает — он по умолчанию переопределяет стандартный браузерный поиск и ищет по всем данным, а не только по видимому DOM.

**"$mol — это же только для SPA? Для сайтов с интерактивными элементами — бери SSG + веб компоненты."**

Вопрос приоритетов. Для SSG я бы всё равно выбрал $mol — темизация, локализация и адаптивная вёрстка идут из коробки. Что до WC — если нужно воткнуть компонент в существующий проект, то конечно, почему нет. Я бы даже попробовал подключить один в своё $mol-приложение.

**"Где скрытые результаты бенчмарка, которые были плохи для $mol?"**

Начальный бенчмарк был настроен неправильно — в корне были мусорные файлы, которые подтягивались во все бандлы локально. Реальный размер оказался ~185 KB. Честно отметил это в [бенче](https://github.com/b-on-g/todomvc-compare). Можете проверить, собрав сами. Буду рад реальному сравнению от независимого автора — так и рождается истина, по-научному.
